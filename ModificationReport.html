詳細修改說明

1. 通訊線程立即停止：阻塞 → 非阻塞狀態機
CommThread.cs — ProcessImmediateStops()

問題：DoImmediateStopIfRequested() 內含 Thread.Sleep(tDec * 1000 + 50)，阻塞通訊線程約 550ms，期間無法處理緊急停止、狀態輪詢或看門狗。

修正方式：引入 StopPhase 狀態機，將三步驟（VelocityChange → 等待 → Sd_Stop）拆為非阻塞迭代。

+ private enum StopPhase { None, Decelerating }
+ private readonly StopPhase[] _stopPhase = new StopPhase[AXIS_COUNT];
+ private readonly long[] _stopDeadlineTicks = new long[AXIS_COUNT];

- Thread.Sleep(waitMs);  // 阻塞 550ms
+ // 非阻塞：每次主迴圈檢查 deadline 是否到達
+ if (DateTime.UtcNow.Ticks >= _stopDeadlineTicks[i])
+     _ecat.CS_ECAT_Slave_Motion_Sd_Stop(...);

效果：通訊線程在等待減速期間仍能每 10ms 處理緊急停止、警報偵測與看門狗心跳。

2. DLL 回傳碼檢查機制
CommThread.cs — DoPollStatus()

問題：4 個 DLL 輪詢呼叫的回傳碼被反覆覆寫而從未檢查。硬體通訊中斷時使用舊值繼續運行。

修正方式：每個 DLL 呼叫使用獨立變數 r1~r4。任一失敗則累計該軸錯誤次數，連續 5 次觸發全軸安全停止。

+ private const int POLL_ERROR_THRESHOLD = 5;
+ private readonly int[] _pollErrorCount = new int[AXIS_COUNT];

- ret = _ecat...Get_Actual_Position(...);
- ret = _ecat...Get_Current_Speed(...);
+ var r1 = _ecat...Get_Actual_Position(...);
+ var r2 = _ecat...Get_Current_Speed(...);
+ if (r1 != 0 || r2 != 0 || r3 != 0 || r4 != 0) {
+     _pollErrorCount[i]++;
+     if (_pollErrorCount[i] >= POLL_ERROR_THRESHOLD)
+         _sdStopAllRequested = true;
+     continue;
+ }
3. 鎖順序統一 + _queueLength 跨鎖修正
CommThread.cs — DoPollStatus() / UpdateQueueLengths()

問題 A：DoPollStatus → CheckAutoSdStop 的鎖順序為 _stateLock → _queueLock（ABBA 風險）。

問題 B：_queueLength[] 在 _queueLock 下寫入，在 _stateLock 下讀取。

修正方式：

  • CheckAutoSdStop 呼叫移至 _stateLock 外部（使用 needAutoSdStop[] 延遲標記）
  • UpdateQueueLengths 內部加入 lock (_stateLock)，鎖順序統一為 _queueLock → _stateLock
修正後全域鎖順序：
  ProcessQueues:    _queueLock → _stateLock  ✓
  UpdateQueueLengths: _queueLock → _stateLock  ✓
  DoPollStatus:     _stateLock only (不嵌套)    ✓
  CheckAutoSdStop:  _queueLock only (不嵌套)    ✓
4. ProcessCardStateChange 原子性保護
CommThread.cs — 新增 _stateChangeLock

問題：通訊線程寫入 _requestedCardState + _cardStateChangeRequest，主線程在 ProcessCardStateChange 中讀取並清除。三步非原子操作可能遺失狀態變更。

修正方式：引入 _stateChangeLock，所有讀寫 _cardStateChangeRequest / _requestedCardState 的位置均加鎖。

+ private readonly object _stateChangeLock = new();

// 通訊線程寫入（DoPollStatus、DoRalm、CheckMainThreadWatchdog）
+ lock (_stateChangeLock) {
      _requestedCardState = CardState.ALARM;
      _cardStateChangeRequest = true;
+ }

// 主線程讀取
  public CardState ProcessCardStateChange() {
+     lock (_stateChangeLock) {
          if (_cardStateChangeRequest) { ... }
+     }
  }
5. RobotLogger WriterLoop 信號競態修正
Robot.Core/Logging/RobotLogger.cs — WriterLoop()

問題：Wait() 返回與 Reset() 之間存在信號丟失窗口，可能導致日誌延遲最多 500ms。

修正方式：重構迴圈為「先 Flush → 確認 Queue 為空才 Wait」模式，消除信號丟失窗口。

- _hasEntries.Wait(500ms);
- _hasEntries.Reset();
- FlushQueue();
+ FlushQueue();
+ if (_queue.IsEmpty)
+     _hasEntries.Wait(500ms);
+ _hasEntries.Reset();

效果：即使信號在 Wait 返回後被 Reset 清除，已經在 FlushQueue 中處理了所有待寫項目。下一輪迭代會立即再次 FlushQueue，不會有延遲。

6. MonitorServer 遷移至 Robot.Motion.RA605
遷移 Robot.Driver.Delta → Robot.Motion.RA605

問題：MonitorServer 位於 Driver 層，只能存取低階軸卡資料，無法提供末端位姿等高階資訊。

修正方式：

  • 將 MonitorServer.cs 從 Robot.Driver.Delta 搬至 Robot.Motion.RA605
  • 命名空間從 Robot.Driver.Delta 改為 Robot.Motion.RA605
  • 建構子依賴從 IAxisCard 改為 IMotionController
  • 嵌入式資源（monitor.html + meshes）隨之搬遷
  • BuildStateJson 新增末端位姿資料（endEffectorPosition + endEffectorPosture）

- MonitorServer(IAxisCard driver, ...)

// 遷移後
+ namespace Robot.Motion.RA605

7. BroadcastLoop 從 sync-over-async 改為全非同步
Robot.Motion.RA605/MonitorServer.cs — BroadcastLoopAsync()

問題：ws.SendAsync(...).GetAwaiter().GetResult() 在 Thread 上同步等待，若客戶端網路慢則阻塞整個廣播迴圈。

修正方式：

  • 廣播迴圈改為 async Task BroadcastLoopAsync()，透過 Task.Run 啟動
  • 使用 Task.WhenAll 並行發送至所有客戶端
  • 每個客戶端附帶 1 秒逾時保護（CancellationTokenSource.CreateLinkedTokenSource + CancelAfter）
  • 廣播間隔改用 await Task.Delay 取代 Thread.Sleep
- private Thread? _broadcastThread;
+ private Task? _broadcastTask;

- ws.SendAsync(segment, ...).GetAwaiter().GetResult();
+ await Task.WhenAll(snapshot.Select(ws => SendWithTimeout(ws, segment, ct)));

+ private async Task SendWithTimeout(WebSocket ws, ...) {
+     using var timeoutCts = CancellationTokenSource.CreateLinkedTokenSource(ct);
+     timeoutCts.CancelAfter(1000);
+     await ws.SendAsync(data, ..., timeoutCts.Token);
+ }

效果：多個客戶端的發送並行執行，單一慢客戶端不會阻塞其他客戶端或整個廣播週期。逾時後自動移除失效連線。

修正後鎖策略總結

修正後所有程式碼路徑的鎖取得順序已統一，消除 ABBA 死鎖風險：

鎖物件用途持有者
_stateLock保護 pos/speed/state/queueLength 陣列通訊線程寫、主線程讀
_queueLock保護指令佇列主線程入隊、通訊線程出隊
_stateChangeLock保護 cardState 變更請求的原子性通訊線程寫、主線程讀
_immediateStopLock保護立即停止旗標主線程寫、通訊線程讀

巢狀鎖取得順序（全域唯一方向）

_queueLock → _stateLock     (ProcessQueues, UpdateQueueLengths)
_stateChangeLock             (單獨使用，不與其他鎖巢狀)
_immediateStopLock           (單獨使用，不與其他鎖巢狀)

備註與後續建議

  • MonitorServer BuildStateJson 仍觸發多次 RefreshState：每個 IAxisCard 屬性存取都會觸發一次 RefreshState()。由於通訊線程以 10ms 週期統一更新所有軸資料，各屬性間的時間差異極小（微秒級）。對於 20Hz 的監控用途，此行為可接受。如需進一步優化，可在 IAxisCard 介面新增 GetSnapshot() 方法。
  • 前端適配：若 monitor.html 前端有解析 WebSocket JSON 的程式碼，需確認其能忽略新增的 endEffectorPosition / endEffectorPosture 欄位，或加以利用。
  • Demo.EndEffector 未測試：該專案未包含在 Robot.sln 中，若有使用 MonitorServer 的程式碼需同步更新 import 路徑。
