5. 看門狗場景分析（全場景）

5.1 場景一覽

CRITICAL 主程序 crash（未處理例外） CommService 管道斷開（IOException） < 100ms SdStopAll → 等待停止 → End → 自行退出
CRITICAL 主程序被 kill（Task Manager） CommService 管道斷開 + PID 不存在 < 100ms 同上

5.2 各場景詳細流程

場景 A：主程序 Crash / 被 Kill

這是此架構要解決的首要問題。

  1. 主程序發生未處理例外或被 Task Manager 終止
  2. OS 自動關閉主程序持有的 Named Pipe Client 端
  3. CommService 的 CommandHandler.ReadAsync() 收到 IOException（管道斷開）
  4. WatchdogMonitor 接收斷開通知，立即執行：
    1. CommThread.RequestSdStopAll() → 全軸減速停止（0.5 秒減速）
    2. 等待 1 秒確認各軸 Mdone
    3. CommThread.RequestEnd() → Servo OFF、Master Close
    4. CommService 程序安全退出

最差反應時間： 管道偵測 100ms + 減速 500ms = 600ms 內全部停止

時間軸：
 0ms    主程序 crash
 ~50ms  OS 清理管道 → CommService IOException
 ~60ms  WatchdogMonitor 觸發 SdStopAll
 ~560ms 全軸減速完成（TDec=0.5s）
 ~1560ms Servo OFF + Master Close
 ~1600ms CommService 退出
場景 B：主程序 Deadlock / 卡死
  1. 主程序的 UI 或業務邏輯卡死（deadlock、無窮迴圈）
  2. PipeAxisCard 的心跳 Timer 可能也停止（取決於卡死的線程）
  3. CommService 在 2 秒內未收到任何心跳訊息
  4. WatchdogMonitor 心跳逾時觸發：
    1. CommThread.RequestSdStopAll()
    2. CardState 設為 ALARM（透過狀態推送通知主程序——如果主程序還能處理的話）
    3. 不會自動 End/退出（因為管道可能仍然連接；等待主程序恢復或手動處理）

最差反應時間： 2 秒心跳逾時 + 0.5 秒減速 = 2.5 秒內全部停止

與現有架構的差異： 現有架構在此場景下反應時間相同（CommThread 內部看門狗也是 2 秒），但改為獨立程序後，即使主程序完全無回應，CommService 依然能獨立完成停止操作。

場景 C：CommService Crash

CommService 自身 crash 的處理——硬體層面。

  1. CommService 發生未處理例外（如 DLL callback 異常）
  2. 硬體行為：
    • EtherCAT Master 隨程序關閉而斷開
    • Delta 驅動器偵測到 Master 離線 → 驅動器內建保護觸發 → 驅動器自行停止馬達
    • 這是硬體層的最後一道防線，與軟體看門狗無關
  3. 主程序行為：
    1. PipeAxisCard 的狀態接收線程收到 IOException
    2. IsCommAlive 回傳 false
    3. DeltaDriver.HeartbeatCallback 偵測到 → 記錄 Fatal 日誌
    4. 可選：CommServiceLauncher 嘗試重啟 CommService（需重走 Start → Initial 流程）

反應時間： < 200ms（主程序下一次心跳回調時發現）

重要：CommService crash 時馬達是否安全？
  • Real 模式：安全。EtherCAT 協定本身有 watchdog，Master 離線後驅動器自行觸發安全停止。
  • Mock 模式：無實體硬體，無需擔心。
  • 建議：CommService 內部對所有 DLL 呼叫包 try-catch，在 crash 前盡量執行 DoEstop() + DoEnd()。
場景 D：Named Pipe 連線異常中斷（非程序 crash）

例如：OS 資源不足導致管道 handle 被回收（極罕見）。

  1. 雙方同時收到 IOException
  2. CommService 端：WatchdogMonitor 觸發 SdStopAll → End → 退出
  3. 主程序端：PipeAxisCard 報告 IsCommAlive = false → Fatal 日誌

行為與「場景 A：主程序 crash」的 CommService 端處理一致。

序列化結論

最大的序列化挑戰是 MoveMultiAxisPVT 的二維陣列。以 10 個 PVT 點、6 軸計算：

  • MultiTargetPos: 6 × 10 = 60 個 int
  • MultiTargetTime: 6 × 10 = 60 個 int
  • 總 Payload 約 1.2KB JSON，序列化時間 < 0.1ms
  • Named Pipe 本地傳輸 < 0.05ms
  • 對 10ms 循環週期的影響可忽略
