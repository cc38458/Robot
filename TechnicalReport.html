1. 專案概述與架構評估

本專案為 RA605 六軸機械臂的高階控制框架，採用分層架構，支援 Real（EtherCAT 硬體）與 Mock（模擬）兩種後端模式，並提供唯讀 WebSocket 監控介面。

架構分層

┌──────────────────────────────────────────────────────┐ │ Demo.StepConsole / 上位應用程式 │ ├──────────────────────────────────────────────────────┤ │ Robot.Motion.RA605（高階運動控制） │ │ RA605RobotApp ← MotionController ← RA605Kinematics │ ├──────────────────────────────────────────────────────┤ │ Robot.Driver.Delta（驅動層） │ │ DeltaDriver ← CommThread ← IEtherCatApi │ │ MonitorServer（WebSocket 監控） │ ├──────────────────────────────────────────────────────┤ │ Robot.Core（核心介面與模型） │ │ IAxisCard / IMotionController / Enums / Models │ └──────────────────────────────────────────────────────┘

整體評價

專案整體架構清晰，分層合理，介面設計良好。核心功能（EtherCAT 通訊、正逆運動學、PVT 軌跡規劃）的實作品質中上。以下是逐一檢查後發現的各類問題。

3. 嚴重邏輯問題（需立即修正）

嚴重DoImmediateStopIfRequested 阻塞通訊線程
CommThread.cs:867-900

DoImmediateStopIfRequested() 方法中包含 Thread.Sleep(waitMs)， 當某軸要求立即停止時，整條通訊線程會被阻塞 tDec * 1000 + 50 毫秒（預設約 550ms）。 在此期間：

  • 狀態輪詢完全中斷 — 其他軸的位置、速度、警報無法更新
  • 緊急停止請求無法被處理 — 若同時有其他軸觸發警報，通訊線程無法回應
  • 看門狗心跳停止更新 — 可能觸發假看門狗超時

建議修正：將立即停止改為非阻塞式狀態機，在主迴圈每次迭代中推進狀態，而非使用 Thread.Sleep 等待。

// 建議：每軸使用停止狀態追蹤，而非阻塞等待
enum StopPhase { None, Decelerating, WaitingSdStop }
private readonly StopPhase[] _stopPhase = new StopPhase[AXIS_COUNT];
private readonly long[] _stopStartTick = new long[AXIS_COUNT];

// 在主迴圈中非阻塞式推進
private void AdvanceImmediateStop(ushort axis) {
    if (_stopPhase[axis] == StopPhase.Decelerating) {
        var elapsed = (DateTime.UtcNow.Ticks - _stopStartTick[axis]) / TimeSpan.TicksPerMillisecond;
        if (elapsed >= _immediateStopTDec[axis] * 1000 + 50) {
            _ecat.CS_ECAT_Slave_Motion_Sd_Stop(...);
            _stopPhase[axis] = StopPhase.None;
        }
    }
}
嚴重DoPollStatus 中 DLL 回傳值被靜默忽略
CommThread.cs:590-649

DoPollStatus() 方法對每個軸連續呼叫 4 個 DLL 函式，但回傳碼 ret 被反覆覆寫而從未檢查： 

ret = _ecat.CS_ECAT_Slave_Motion_Get_Actual_Position(...);  // ret 未檢查
ret = _ecat.CS_ECAT_Slave_Motion_Get_Current_Speed(...);    // 覆寫上一個 ret
ret = _ecat.CS_ECAT_Slave_Motion_Get_Mdone(...);            // 再次覆寫
ret = _ecat.CS_ECAT_Slave_Motion_Get_StatusWord(...);       // 最終 ret 也未使用

若任一 DLL 呼叫失敗（如硬體通訊中斷），系統會使用未更新的舊值或初始值（0）繼續運行， 可能導致位置讀數錯誤或遺漏警報偵測。在硬體實機上，這可能造成碰撞風險。

建議修正：檢查每個回傳碼，累計錯誤次數，連續失敗達到閾值時觸發安全停止。

嚴重MonitorServer.BuildStateJson 多次觸發 RefreshState 造成狀態不一致
MonitorServer.cs:239-256

BuildStateJson() 在 20Hz 廣播時依序存取 _driver.Pos、_driver.Speed、 _driver.State、_driver.QueueLength、_driver.AxisCardState。 每次屬性存取都會觸發 DeltaDriver.RefreshState()，意味著：

  • 每次廣播觸發 5 次 lock + 5 次狀態複製，效能浪費
  • pos 與 speed 可能來自不同時刻的快照，在快速運動中會出現 pos/speed 不匹配
  • 廣播線程與主線程同時頻繁呼叫 RefreshState，造成不必要的鎖競爭

建議修正：新增一個一次性讀取全部狀態的方法，或在 BuildStateJson 中只呼叫一次 RefreshState 後使用快取值。

4. 執行緒安全與併發問題

高風險_queueLength 陣列跨鎖寫讀不一致
CommThread.cs:264-268 / CommThread.cs:102-111

UpdateQueueLengths() 在 _queueLock 保護下寫入 _queueLength[]， 但 GetState() 在 _stateLock 保護下讀取同一陣列。 兩個不同的鎖保護同一資料的讀寫，可能讀到半更新的中間狀態。 

// 寫入端（受 _queueLock 保護）
private void UpdateQueueLengths() {
    for (int i = 0; i < AXIS_COUNT; i++)
        _queueLength[i] = _queues[i].Count;  // ← 寫在 _queueLock 內
}

// 讀取端（受 _stateLock 保護）
public void GetState(..., int[] queueLen) {
    lock (_stateLock) {
        Array.Copy(_queueLength, queueLen, AXIS_COUNT);  // ← 讀在 _stateLock 內
    }
}

建議修正：將 _queueLength 的更新也放入 _stateLock 內，或在 GetState 中同時取鎖。

高風險_cardState 跨線程讀寫的一致性風險
CommThread.cs:34-36 / CommThread.cs:950-958

_cardState 宣告為 volatile，但 ProcessCardStateChange() 方法執行的是「先讀 flag → 寫 state → 清 flag」三步組合操作，並非原子性。 通訊線程可能在主線程讀取 _cardStateChangeRequest 之後、清除之前設定新的狀態請求， 導致該請求被遺失。 

// 非原子操作：通訊線程可能在步驟 1 和 3 之間寫入新的請求
public CardState ProcessCardStateChange() {
    if (_cardStateChangeRequest)          // 步驟 1：讀取 flag
    {
        _cardState = _requestedCardState; // 步驟 2：套用
        _cardStateChangeRequest = false;  // 步驟 3：清除 ← 此時通訊線程可能已寫了新 request
    }
    return _cardState;
}

建議修正：使用 Interlocked.CompareExchange 或加鎖確保原子性，或改用 ConcurrentQueue 傳遞狀態變更。

高風險ProcessQueues 與 DoPollStatus 的鎖取得順序不一致
CommThread.cs:682-728 / CommThread.cs:605-676

雖然這兩個方法目前在同一條通訊線程上依序執行，不會真正死鎖， 但它們呈現經典的 ABBA 鎖順序，屬於潛在危險：

方法 鎖取得順序
ProcessQueues() _queueLock → _stateLock
DoPollStatus() → CheckAutoSdStop() _stateLock → _queueLock

若未來重構將任一方法移至不同線程（例如將狀態輪詢獨立出來）， 將立即產生死鎖。建議統一鎖取得順序或重新設計鎖策略。

高風險BroadcastLoop 使用 sync-over-async 反模式
MonitorServer.cs:219-220

ws.SendAsync(...).GetAwaiter().GetResult() 在 Thread 上同步等待非同步操作。 當連線客戶端網路較慢或有多個客戶端時，單次廣播可能耗時數百毫秒，導致：

  • 廣播頻率降低（無法維持 20Hz）
  • 若某客戶端斷線但 socket 尚未關閉，SendAsync 可能長時間阻塞

建議修正：改為 async 廣播迴圈，或使用 Task.WhenAll 並行發送並加上超時取消。

高風險RobotLogger.WriterLoop 存在信號丟失窗口
RobotLogger.cs:106-117

在 WriterLoop 中： 

_hasEntries.Wait(TimeSpan.FromMilliseconds(500)); // ← 收到信號返回
_hasEntries.Reset();   // ← 此處與上一行之間，若另一線程 Enqueue + Set()
FlushQueue();          //    該次 Set() 的信號被 Reset 清除，延遲最多 500ms

此問題不會造成日誌遺失，但在高頻寫入場景下，日誌可能出現最多 500ms 的延遲。 對於即時除錯場景（如追查碰撞原因），此延遲可能影響日誌時序分析。

建議修正：改用 AutoResetEvent 或 BlockingCollection 取代手動信號管理。

5. 架構耦合與設計問題

中等RA605RobotApp 直接依賴具體實作類別
RA605RobotApp.cs:18-19

入口類別 RA605RobotApp 直接建立並持有 DeltaDriver（具體類） 與 MotionController（具體類），而非依賴 IAxisCard 與 IMotionController 介面。 這違反了依賴反轉原則（DIP）。 

// 目前
private readonly DeltaDriver _driver;        // ← 具體類別
private readonly MotionController _motion;   // ← 具體類別

// 建議
private readonly IAxisCard _driver;
private readonly IMotionController _motion;

影響：無法在不修改 RA605RobotApp 的情況下替換驅動或運動控制實作， 也無法進行單元測試時注入 mock 物件。

中等MotionCommand 為臃腫的「萬用」類別
MotionCommand.cs:1-100

MotionCommand 使用單一類別搭配 nullable 欄位承載 7 種不同指令類型的資料。 不同指令實際使用的欄位不同，容易出現：

  • 忘記填寫某類型專用欄位 → 運行時使用預設值 0，產生難以追蹤的邏輯錯誤
  • 無法在編譯期確認指令資料的完整性
指令類型 實際使用欄位 未使用但存在的欄位
MoveAbsolute Axis, Dist, StrVel, ConstVel, EndVel, TAcc, TDec NewTargetSpd, TSec, DataCount, TargetPos, TargetTime, Multi*
VelocityChange Axis, NewTargetSpd, TSec Dist, StrVel, ConstVel, EndVel, TAcc, TDec, DataCount, Multi*
MultiAxisPVT Multi* Axis, Dist, StrVel, ConstVel, EndVel, TAcc, TDec, NewTargetSpd, TSec

建議修正：使用 record + discriminated union 模式，或至少使用工廠方法確保建構正確性。

中等MoveRelativeEndEffector 與 ContinuousMoveLoop 存在大量重複的旋轉矩陣建構程式碼
MotionController.cs:300-315 / MotionController.cs:428-453

兩個方法中都包含完全相同的「從 Yaw/Pitch/Roll 增量建構旋轉矩陣並疊加到當前姿態」邏輯， 約 15 行程式碼完全重複。應提取為共用方法。 

// 重複出現的程式碼（兩處完全一致）
var rotInc = Matrix4x4.CreateRotationZ(yawDeg * MathF.PI / 180f)
           * Matrix4x4.CreateRotationY(pitchDeg * MathF.PI / 180f)
           * Matrix4x4.CreateRotationX(rollDeg * MathF.PI / 180f);
var curRot = new Matrix4x4(
    currentPosture.M11, currentPosture.M12, currentPosture.M13, 0,
    ...);
var newRot = rotInc * curRot;
// ... 手動抄寫 9 個矩陣元素 ...

建議修正：提取為 ApplyRotationIncrement(Matrix4x4 current, float yawDeg, float pitchDeg, float rollDeg, float newX, float newY, float newZ) 方法。

中等MonitorServer 位於 Driver 層但需要高階資訊
MonitorServer.cs / RA605RobotApp.cs:242

MonitorServer 屬於 Robot.Driver.Delta 專案，但被 RA605RobotApp 建構時傳入 _driver（IAxisCard）。監控只能取得低階的軸角度資料， 無法取得末端位姿（EndEffectorPosition）等高階資訊，限制了監控功能的完整性。

建議修正：將 MonitorServer 移至 Robot.Motion.RA605 層，或定義一個 IRobotStateProvider 介面供監控使用。

中等缺乏運動參數驗證
DeltaDriver.cs / MotionController.cs

多數運動控制方法僅驗證軸卡狀態與軸號，但未驗證運動參數的合理性：

  • 速度：未檢查是否為負值或超過硬體最大速度限制
  • 加減速時間：MoveAbsolute 等方法未驗證 tAcc、tDec 是否 > 0
  • 角度範圍：軸空間運動未檢查目標角度是否在關節限位內（RA605Kinematics 中有限位定義但軸空間方法未使用）
  • PVT 時間：未驗證 targetTime 陣列是否遞增

建議修正：在 DeltaDriver 或 MotionController 層加入參數範圍驗證，至少記錄警告。

中等缺乏單元測試專案
Robot.sln

整個解決方案無任何測試專案。對於控制實體機械臂的軟體，至少以下部分應有單元測試覆蓋：

  • RA605Kinematics：正逆運動學的精度驗證、奇異點處理、關節限位檢查
  • MotionController：PVT 軌跡點產生的正確性、加速度限制邏輯
  • CommThread 狀態機：各狀態轉移的正確性
  • MotionCommand 建構：各類型指令的欄位完整性

6. 命名規範與程式碼風格

低CardState.CONNCET 拼字錯誤
CardState.cs:14

CONNCET 應為 CONNECT。此拼字錯誤貫穿整個程式碼庫（約 15 處引用）， 雖然不影響功能，但會降低程式碼可讀性，且對外部開發者造成困惑。

低持續移動速度變數命名過於簡短
MotionController.cs:32

_cdx、_cdy、_cdz、_cdYaw、_cdPitch、_cdRoll 命名不直覺。建議改為更具描述性的名稱： 

// 目前
private float _cdx, _cdy, _cdz, _cdYaw, _cdPitch, _cdRoll;

// 建議
private float _targetVelX, _targetVelY, _targetVelZ;
private float _targetVelYaw, _targetVelPitch, _targetVelRoll;
低Dist 參數名稱語意不明
IAxisCard.cs:105-106

MoveAbsolute 的參數名為 dist，但對於絕對移動而言，此值代表「目標位置」而非「距離」。 建議 MoveAbsolute 使用 targetPos，MoveRelative 使用 delta 或 distance。

低枚舉值使用全大寫命名
CardState.cs / MotorState.cs

CardState.NULL、MotorState.STOP 等使用全大寫命名。 C# 慣例建議枚舉值使用 PascalCase（如 CardState.Null、MotorState.Stop）。 但考慮到此專案涉及硬體控制且全大寫在嵌入式領域常見，此為次要問題。

低方法名稱 Initial() 不符慣例
IAxisCard.cs:58

Initial() 應為 Initialize()。Initial 是形容詞， Initialize 才是動詞。這在 RA605RobotApp 中已修正為 Initialize()， 但底層介面與實作仍為 Initial()，造成不一致。

低Ralm / Estop 等縮寫缺乏文件說明
IAxisCard.cs:69-76

Ralm（Reset Alarm）和 Estop（Emergency Stop）為領域縮寫， 對非硬體背景的開發者不直覺。建議至少在介面的 XML 文件中補充完整英文名稱。 或考慮更直覺的命名如 EmergencyStop()、ResetAlarm()。

低Thread.Sleep 硬編碼散落各處
DeltaDriver.cs:124 / DeltaDriver.cs:163

DeltaDriver.End() 中 Thread.Sleep(500) 和 Ralm() 中 Thread.Sleep(200) 為硬編碼魔術數字。建議提取為具名常數並記錄用途。

低BATCH_PERIOD_SEC 計算公式難以理解
MotionController.cs:23
private const float BATCH_PERIOD_SEC = CONTINUOUS_CYCLE_MS * PVT_POINTS_PER_BATCH / 2 / 1000f;
// = 50 * 10 / 2 / 1000 = 0.25 秒

為何除以 2？這個「半個批次週期」的物理意義未在註解中說明。 建議加上說明或拆解為更易讀的計算方式。

7. 品質與可維護性

資訊RA605RobotApp.CalibrateZero 與 CommThread.RequestCalibrateZero 重複檢查 Mock 模式
RA605RobotApp.cs:218-226 / CommThread.cs:168-184

Mock 模式的檢查在兩處都執行了： 

// RA605RobotApp.CalibrateZero()
if (BackendMode == RobotBackendMode.Mock) { ... return true; }

// CommThread.RequestCalibrateZero()
if (_isMockBackend) { ... return true; }

應統一在一處檢查，避免未來修改時遺漏其一。

資訊Mock 後端的 Get_Actual_Position 映射不一致
EtherCatApiAdapter.cs:110

MockEtherCatApi.CS_ECAT_Slave_Motion_Get_Actual_Position 委派至 CEtherCAT_Mock.CS_ECAT_Slave_Motion_Get_Position（非 Get_Actual_Position）。 雖然在 Mock 中兩者返回相同值，但映射不一致可能在未來擴展 Mock 功能時引起混淆。

資訊EtherCAT_DLL_Mock 使用 static 狀態，無法支援多實例
EtherCAT_DLL_Mock.cs:12-33

CEtherCAT_Mock 所有狀態皆為 static 欄位。 若未來需要在同一進程中模擬多張軸卡或進行平行測試，靜態狀態會互相干擾。 此設計與 Real DLL（全域狀態的 P/Invoke）一致，因此對單機使用無影響， 但會限制自動化測試的平行執行。

資訊Dispose 模式不完整
DeltaDriver.cs:363-371 / RA605RobotApp.cs:284-296

DeltaDriver.Dispose() 未呼叫 End()，可能在未正確關閉 EtherCAT 連線的情況下釋放資源。 而 RA605RobotApp.Dispose() 呼叫了 _driver.End() 再呼叫 _driver.Dispose()， 若使用者直接 Dispose DeltaDriver 而非 RA605RobotApp，Servo 可能不會被關閉。

資訊持續移動迴圈使用 volatile bool 而非 CancellationToken
MotionController.cs:30

_continuousRunning 使用 volatile bool 控制線程生命週期， 但 .NET 推薦使用 CancellationToken 模式，可提供更好的取消傳播與異常處理。
